PP-OCRv6 合成数据与业务 Benchmark 对比说明

对比第一版 all-overlay 合成训练和 V2 字幕/水印 + hard negative 训练，说明为什么合成验证集表现正常，但真实业务 benchmark 召回仍不达标。

一句话结论

合成数据路线可以把 PP-OCRv6 训练收敛，但当前合成分布还没有覆盖真实业务视频的字幕、水印、小目标、底部区域和压缩退化。第一版 recall 较高但误检严重；V2 hard negative 显著降低误检，但模型变保守，真实业务 recall 下降。

第一版 all-overlay

business test recall 49.30%，但 clean FP image rate 75.76%，误擦 native/商品文字风险很高。

V2 subtitle/watermark + hard negative

clean FP image rate 降到 10.10%，但 business test recall 只有 28.05%。

下一步重点

不要只继续加 epoch，应调整数据分布、真实退化、9:16 版式、小目标采样，并在业务 val 上做阈值扫描。

两版在 synthetic val 上都能达到较高 recall，但迁移到真实业务 test 后明显下降。V2 的 clean FP 明显改善，证明 hard negative 有价值；但业务 recall 更低，说明正样本合成难度和分布还不够。

为什么合成数据在业务 Benchmark 表现不佳

  • 真实字幕/水印样式覆盖不足。真实视频存在压缩毛边、运动模糊、低清重采样、复杂描边、阴影、半透明、平台水印、用户名水印等，本轮合成退化还不够。
  • 9:16 版式和训练 crop 不匹配。业务视频大多是竖屏，当前训练仍使用 640x640 RandomCrop，容易破坏底部字幕、角落水印和边缘小目标的空间分布。
  • 小目标召回差。错误归因显示小水印、小字幕、小符号是主要漏检来源，尤其是 watermark 和底部元素。
  • hard negative 让模型更保守。V2 clean FP 从 75.76% 降到 10.10%，但 overlay recall 从 49.30% 降到 28.05%，说明负样本有效但正样本难例密度不足。
  • IoU 指标不能完全代表擦除质量。框略大可能利于擦除，但 IoU 会惩罚大框。后续需要补 coverage recall，衡量 GT overlay 是否被完整覆盖。

后续数据集调整

方向 调整内容 目标
真实业务风格正样本 补小字号底部字幕、多行字幕、描边/阴影字幕、低透明角落水印、平台/达人水印、用户名水印、压缩模糊退化 提升 subtitle/watermark recall
画幅与尺度 按 9:16 完整帧合成，减少正方形 crop 对版式的破坏；提高输入尺度或增加多尺度训练 改善底部字幕、角落水印、小目标
hard negative 比例 保留 hard negative，但降低压制强度或提高难正样本比例，避免模型只学会少出框 在 clean FP 可控前提下找回 recall
阈值与后处理 在业务 val 上扫描 box_thresh 0.10/0.15/0.20/0.25/0.30/0.35，同时观察 unclip_ratio 找到业务工作点，而不是只追 synthetic hmean
评估指标 保留 IoU@0.5 recall，同时新增 coverage@0.90/0.95 recall、clean FP、native overlap 更贴近擦除“无残留、少误擦”的目标

汇报口径

当前实验已经证明训练链路可跑通，PP-OCRv6 能在合成验证集上收敛，但合成数据到真实业务 benchmark 存在明显 domain gap。第一版主要问题是误检 native 内容，V2 通过 hard negative 显著降低误检，但召回过低。下一阶段重点应从“继续训练现有合成集”转向“按真实业务字幕/水印分布重构数据集 + 业务 val 阈值扫描 + coverage 指标”。
