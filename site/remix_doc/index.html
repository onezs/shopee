Creative Video Remix · Meeting Brief

Video Remix Pipeline 项目说明

一个面向跨境电商素材的多阶段 Agent 链路：先清理素材，再理解镜头，判断视频能否混剪， 最后按广告叙事结构选镜头、去重、拼接、字幕渲染和质检。

8主流程阶段，从字幕清理到报告输出
7实际 LLM prompt，用于视觉打标、策略判断和文案生成
3成片模式：REMIX / PARTIAL_REMIX / TRANSLATE_ALL
30fpsStage4 CFR 按帧裁切，减少段首闪帧
总览 阶段逻辑 选片与去重 关键阈值 指标计算 Stage1 标签 Prompt LLM 输出 测试模式 结论

链路总览

系统的核心设计是“规则证据 + LLM 编导 + 可解释产物”。LLM 不直接随机决定一切， 而是在结构化镜头标签、视频 pair 分数、候选组合和硬禁区的基础上做广告编导判断。

Stage0字幕/水印清理
Stage1切镜头 + 视觉打标
Stage2视频级成片策略
Stage3镜头编排 + 口播
Stage4CFR 裁切拼接
Stage4.5Remotion 字幕
Stage5QA 质检
Stage6报告与断点

各阶段逻辑

每个阶段都输出可复盘的 JSON 或中间产物，方便定位问题：是 OCR、打标、视频组合、镜头去重，还是最终拼接。

Stage0 字幕清理

OCR 检测硬字幕、水印、顶部花字，生成 mask 后交给 inpainting backend。

  • LaMa：快，适合快速基线
  • DiffuEraser：质量更稳但慢
  • ProPainter：字幕/水印小 mask 的快方案

Stage1 镜头理解

切分视频镜头，对关键帧或 clip fallback 调视觉 LLM，输出结构化标签。

  • 质量、景别、内容类型
  • 动作、状态、叙事阶段
  • 卖点、可见文字、去重签名

Stage2 策略规划

先用规则层算视频 pair 兼容，再让 LLM 基于证据决定成片模式。

  • 商品语义重叠
  • 视觉/背景/人物一致性
  • 硬禁区不可被 LLM 覆盖

Stage3 脚本生成

主视频骨架优先，辅助视频只补缺口；按 AIDA 结构选镜头、去重并生成口播。

  • HOOK / DEV / CLIMAX / CTA
  • 开箱最多一个
  • 重复镜头不再补回

Stage4 混剪组装

先转 CFR 30fps 中间源，再按帧号 trim，降低 VFR 和 GOP seek 带来的闪帧。

  • stage4_cfr_sources
  • edge_guard_frames=3
  • 失败回退按秒裁切

Stage4.5 字幕渲染

Remotion 渲染口播字幕和原字幕翻译回填；水印类文本擦除但不渲染。

  • 口播字幕
  • OCR 原字幕翻译
  • skip_render 水印过滤

Stage5 质检

检查平台时长、HOOK/CTA、镜头数、脚本结构，输出 violation 和建议。

  • duration_short
  • hook_missing
  • cta_missing

Stage6 报告

汇总最终视频、shots index、stage JSON、KB 引用和 QA log，用于人工复盘。

  • remix_report.md
  • stage*_*.json
  • shots_index.json

视频级：Stage2 决定能不能混

每两个视频先算兼容分，低语义或严重视觉冲突会被排除。

+ style_score * 0.24
+ background_score * 0.18
+ human_score * 0.12
+ type_score * 0.06
+ quality_score * 0.06

token_score < 0.12
=> 商品语义重叠过低
=> hard reject

PARTIAL_REMIX

镜头级：Stage3 决定哪些镜头留下

先定主视频骨架，再用辅助视频补充新的动作、状态、卖点或 CTA 画面。

基础过滤：
quality_score >= 4
content_type 不属于：
BACKGROUND / TRANSITION / EMPTY / OTHER

辅助镜头重复：
similarity >= 0.72 => 丢

同角色重复：
similarity >= 0.88 => 留更优
primary backbone
auxiliary inserts

duplicate_signature

Stage1 输出的稳定短签名，用小写英文和下划线概括商品、动作、状态、背景和手部线索。

clear_case_magsafe_detail_tabletop_hand

如果两个镜头签名完全相同，Stage3 相似度直接给到 0.95。

叙事角色数量

数量不是固定值，而由模板、候选池、去重和时长共同决定。

常见短视频：
HOOK    1
DEV     2-4
CLIMAX  1-2
CTA     1

DEV / CLIMAX fallback

当前角色候选不够时，从其它有效角色候选池借镜头，不从 dropped_shots 捞回重复镜头。

可借范围：
available_shots
selected_video_indices
quality_score >= 4
not dropped_shots

关键阈值

这些阈值是当前链路里最影响结果的规则开关。会上如果要解释“为什么这条视频没混”“为什么这个镜头被丢”，优先看这张表。

位置 规则 含义 结果
Stage2 视频 pair token_score < 0.12 两条视频商品语义重叠过低，可能不是同一商品或卖点不连续。 进入 hard_reject_pairs，LLM 不能选择该 pair。
Stage2 视频 pair score >= 0.48 综合商品语义、风格、背景、人物/手部、内容类型和质量后达到混剪线。 该 pair 可作为 REMIX 或 PARTIAL_REMIX 候选。
Stage2 视频 pair quality_score < 0.60 视频组里较弱一侧的平均质量低于可用线。 降分并加入拒绝原因：素材质量不可用。
Stage2 视频 pair severe_visual_count >= 2 风格、背景、人物/手部三类视觉风险中至少两类严重冲突。 判为严重视觉冲突叠加，pair 不适合混剪。
Stage3 基础过滤 quality_score >= 4 镜头质量至少达到可用线，避免模糊、构图差、主体不清晰素材进入成片。 低于该值的镜头不进入 Stage3 候选池。
Stage3 基础过滤 content_type 排除背景/转场/空镜 BACKGROUND、TRANSITION、EMPTY、OTHER 不承担商品叙事。 直接排除，不参与 HOOK/DEV/CLIMAX/CTA 分配。
Stage3 辅助镜头去重 similarity >= 0.72 辅助视频镜头与主视频同角色镜头高度相似，且没有新增信息量。 丢辅助镜头，记录 duplicate_with_primary_backbone。
Stage3 同角色去重 similarity >= 0.88 同一叙事角色里两个镜头重复度很高。 只保留质量更高、主视频优先或时间更早的一个。
Stage3 签名去重 duplicate_signature 相同 Stage1 认为两个镜头是同一类商品动作/状态/背景组合。 相似度直接设为 0.95，通常会被去重。
Stage3 包装/开箱 包装阶段最多保留 1 个 intro_packaging、unboxing、in_package 都视为包装/开箱阶段。 后续包装镜头丢弃，记录 duplicate_package_or_unboxing_phase。
Stage4 CFR 裁切 edge_guard_frames = 3 按帧裁切时，段首段尾各避开 3 帧。 减少 GOP/VFR 边界带来的黑帧、闪帧和残帧。

指标如何计算

指标分两类：Stage2 是视频级 pair 兼容分，用来判断视频能不能混；Stage3 是镜头级 similarity，用来判断镜头是否重复。

Stage2 视频级兼容分

每两个视频会先提取各自的镜头集合，再计算商品语义、风格、背景、人物/手部、类型和质量分。

pair_score =
  token_score      * 0.34
+ style_score      * 0.24
+ background_score * 0.18
+ human_score      * 0.12
+ type_score       * 0.06
+ quality_score    * 0.06

这个分数只是规则证据。最终 Stage2 LLM 可以覆盖软风险，但不能选择 hard_reject_pairs。

Stage3 镜头相似度

镜头相似度用于去重。签名完全相同时直接高相似，否则按 token、类型、主体、景别和叙事阶段加权。

if duplicate_signature 相同:
  similarity = 0.95

else:
  similarity =
    token_score * 0.42
  + type_score  * 0.18
  + focus_score * 0.10
  + scale_score * 0.05
  + phase_score * 0.25

细节镜头如果同阶段且 token 重叠达到 0.30，相似度最低抬到 0.78，避免重复细节过多。

指标 计算方式 依赖字段 解释
token_score 两个 token 集合的 Jaccard 相似度：交集 / 并集。 description、selling_points、duplicate_signature、visible_text、product_state 衡量两条视频或两个镜头是否在讲同一商品、同一卖点或同一状态。
style_score 先抽取风格 token，再算 Jaccard 相似度。 visual_style、lighting、background、scale、content_type 风格 token 包括 studio、ugc、tabletop、lifestyle、bright、dark、static、motion 等。
background_score 背景/光线 token 的 Jaccard 相似度；如果某视频内部背景不稳定，最高压到 0.35。 background、lighting、description 背景 token 包括 plain、tabletop、home、outdoor、office、store、bright、dark。
human_score 先判断人物/手部一致性，再映射成分数。 human_presence、hand_gender、hand_features、people、model、hands 无人物或一致得高分；女性手/男性手明显冲突会被标记为人物/手部特征不一致。
type_score 两边 content_type 集合的 Jaccard 相似度。 content_type 衡量两条视频是否都包含产品展示、细节、使用演示、包装等相似镜头类型。
quality_score 先算每条视频平均镜头质量，再取较低一侧；归一化为 min_quality / 5，上限 1。 quality_score 确保两条视频里较弱的一条也达到可用质量，不让单条低质视频拖垮成片。
visual_risk_level 综合风格冲突、背景冲突、人物/手部冲突分级。 style_score、background_score、human_consistency 用于提示 LLM 视觉连续性风险。高风险不一定硬拒绝，但必须解释为什么可接受。
dedupe_phase 根据结构化标签归类为 package、detail、usage、product、other。 scene_phase、product_action、product_state、content_type 只有同阶段镜头才更容易被去重。包装/开箱阶段最多保留一个。
focus_score / scale_score 字段完全相同则为 1，否则为 0。 focus_point、scale 辅助判断两个镜头是不是同主体、同景别的重复表达。
解释口径：Stage2 的指标回答“这些视频能不能放在同一条广告里”，Stage3 的指标回答“这些镜头是不是重复表达”。两层分数服务不同，不要混用。

Stage1 标签字段解释

Stage1 的结构化标签是后续所有规则的基础。描述文本给人看，结构化字段给机器排序、去重和判断叙事角色。

字段 含义 典型取值 后续作用
description 一句话描述镜头内容。 手持透明手机壳展示 MagSafe 磁环细节 用于人工复盘，也参与 Stage2/Stage3 语义 token。
content_type 镜头内容大类。 PRODUCT_SHOW、DETAIL、USAGE_DEMO、PACKAGE_SHOW 决定镜头是否可用，以及初始叙事角色。
quality_score 1-10 的画面质量评分。 7、8、9 Stage3 要求 >= 4，同类重复时优先保留高质量镜头。
suitable_for LLM 判断镜头适合的叙事位置。 HOOK、DEV、CLIMAX、CTA 作为旧版角色分配和 fallback 的依据。
narrative_role_scores 每个叙事角色的适配分。 {"HOOK": 0.2, "DEV": 0.8, "CLIMAX": 0.5, "CTA": 0.1} Stage3 优先用它判断镜头归属，而不是只看关键词。
product_action 商品动作。 unboxing、install、hold、closeup、final_display 识别开箱、安装、细节、最终展示，影响排序和去重。
product_state 商品当前状态。 in_package、installed_on_phone、detail_only、final_showcase 帮助判断镜头阶段，避免把开箱或纯细节误选为 CTA。
scene_phase 镜头在商品叙事里的阶段。 intro_packaging、product_reveal、install_or_usage、detail_demo、proof、final_showcase Stage3 排序状态机的核心字段。
selling_points 画面表达的卖点短标签。 magsafe、anti_yellowing、anti_slip、large_capacity 用于判断辅助镜头是否有新增信息量，也参与重复判断。
visible_text LLM 从画面中识别到的短文字。 ["SALE", "MagSafe", "10K"] 补充 OCR 之外的视觉文字线索，参与语义匹配和 CTA 判断。
ocr_text_type 画面文字类型。 title_overlay、subtitle、price_text、watermark 帮助区分标题、字幕、水印、价格，引导 CTA 和字幕渲染策略。
duplicate_signature 稳定去重短签名。 clear_case_magsafe_detail_tabletop_hand 签名相同直接给高相似度；签名 tokens 也参与 Stage2/Stage3 语义判断。
visual_style / background / human_presence 风格、背景和人物/手部线索。 ugc、tabletop bright、hand Stage2 判断视频之间是否视觉连续，避免棚拍/生活方式/不同人物手部硬混。

LLM 输出如何被使用

阶段 LLM 输出 后续作用
Stage1 视觉打标 镜头内容、质量分、叙事角色、动作/状态/阶段、卖点、可见文字、去重签名 支持 Stage2 视频兼容评分，支持 Stage3 镜头排序、去重、CTA 选择和 debug 复盘
Stage2 品类推断 品类名称、推荐景别、内容类型、TTS/音乐偏好 补齐知识库缺失品类，让后续策略可以继续运行
Stage2 SKU 分析 主视频、辅助视频、独立翻译视频、跨视频补充建议 决定主视频骨架和辅助视频插入策略
Stage2 成片模式 REMIX / PARTIAL_REMIX / TRANSLATE_ALL 及候选评估 决定哪些视频进入混剪，哪些视频只做翻译输出

测试模式

test-remix

只做原视频混剪 demo，用于快速看镜头选择和拼接效果。

  • 不做字幕清理
  • 不做翻译视频
  • 不渲染字幕
  • 不生成 TTS/BGM
  • 输出到 output_test_remix/<item>

test-clean

只测试字幕清理 backend，输出可被完整链路断点复用的 clean inputs。

  • 可跑 lama / diffueraser / propainter
  • 输出 clean_inputs/<backend>
  • 输出 stage0_subtitle_clean_*.json
  • 不进入 Stage1-6

会议结论

1. 混剪不是随机拼接。 系统先选主视频骨架，再用辅助视频补充缺失动作、卖点或 CTA 画面。
2. 视频能不能混，主要看商品语义和视觉连续性。 hard_reject_pairs 是不可覆盖硬禁区，LLM 只能在风险 pair 中做编导判断。
3. 镜头能不能留，主要看叙事角色、质量、去重和新增信息量。 重复开箱、重复细节、与主视频高度相似的辅助镜头都会被记录到 dropped_shots。
4. Prompt 的职责边界清晰。 Stage1 理解画面，Stage2 做成片策略，Stage3 做本地化文案；规则代码负责约束和兜底。
5. 工程方向是可解释、可复盘、可测试。 每个阶段都有断点文件和 debug 字段，便于定位问题来自 OCR、打标、成片策略、镜头去重还是拼接。
汇报收尾建议：我们正在把“纯 LLM 判断”收敛成“规则证据 + LLM 编导 + 可解释产物”的工程链路，让视频混剪从效果调参变成可复盘的生产流程。
